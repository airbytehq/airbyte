# Registry-Filtered Agent Connector Docs Implementation Plan

## Overview

Make public agent connector docs respect Sonar registry platform availability so connectors marked experimental are not shown as generally available in docs. The implementation keeps Sonar `registry.json` as the availability source of truth and applies the same enabled-vs-experimental decision to the docs sync workflow, Docusaurus connector catalog, sidebar navigation, and data-replication cross-link banners.

**Domain**: Frontend-only
**Spec mode**: Standard (subagents)
**Recommended /implement mode**: Standard
**Complexity Score**: 65/100
**Devin Delegation**: not-offered

## Implementation Mode Recommendation

Use standard mode. The work is concentrated in Airbyte docs build/workflow code and should be sequenced so the shared availability helper exists before workflow and Docusaurus consumers use it. Team mode would add coordination overhead without clear file-ownership separation.

## Current State

- `agentConnectors.js` reads connector directories only; it has no registry awareness, so any copied directory appears in the `/ai-agents/connectors/` table.
- `AgentConnectorRegistry.jsx` consumes only plugin-provided slugs, so fixing the plugin fixes the visible catalog table without component changes.
- The sync workflow copies all generated connector docs from `airbyte-agent-sdk`, so experimental connector docs keep reappearing unless the workflow filters them.
- Docusaurus autogenerates AI agents sidebar items from the docs tree, so hidden connectors must be removed from sidebar items or omitted from the tree.
- `connectorTypeBanner.js` silently loads an absent JSON file and therefore cannot reliably insert DR-to-agent cross-links.
- Sonar PR 4963 defaults missing `platform_availability` to `enabled`; the docs implementation should match this default unless product explicitly wants fail-closed docs behavior.
- The PR 4963 experimental set at spec time is: `airtable`, `ashby`, `gmail`, `linkedin-ads`, `paypal-transaction`, `shopify`, and `zoho-crm`.

## Architecture

### State ownership
- Canonical availability state: Sonar direct connector registry at `https://connectors.airbyte.ai/registry.json`.
- Derived docs state: enabled slug JSON generated during sync/build, invalidated by the two-hour sync workflow or any forced docs rebuild.
- Fallback: copied docs may be stale; Docusaurus build-time filtering is a safety net over workflow filtering.

### Identity & relationships
- Registry key: `connector_definition_id`; docs key: connector slug from `connector_name` and `docs_url` path.
- Slug normalization must handle hyphenated names exactly as existing docs directories do, for example `paypal-transaction`, `linkedin-ads`, and `zoho-crm`.
- A connector is public-docs-enabled when `platform_availability.state` is absent or `enabled`; it is hidden when state is `experimental`.

### Variants & extensibility
- Future states such as `disabled` or `private_preview` should be treated as hidden unless explicitly allowed. This is intentionally stricter than the missing-state default.
- Future docs may show experimental pages behind an explicit label, but this plan hides them from generally available public listings.
- If registry fetch fails in CI, fail the sync/build step rather than silently publishing an unfiltered list.

### Cross-service coupling
- Airbyte docs build will depend on the Sonar registry CDN. That dependency already exists operationally for platform availability and should be explicit in docs CI logs.
- Docs production freshness depends on the scheduled sync or a Sonar-triggered docs rebuild after registry publish.

### Operational semantics
- Retry: registry fetch should be idempotent and cache to a local generated JSON file during the build.
- Partial failure: do not commit a sync PR that copied unfiltered connector docs after failing to fetch availability.
- Stale state: direct URLs may remain reachable until the next docs deployment if availability changes between scheduled syncs.

## Rollout & Rollback Strategy

- Ship as a focused Airbyte docs PR after Sonar PR 4963 is merged or after the final registry shape is locked.
- Rollback is reverting the Airbyte docs PR; the previous behavior is directory-driven listing.
- If immediate freshness is required, follow up with a Sonar workflow call or repository dispatch that triggers the Airbyte sync/rebuild after registry publish.

## DO NOT (Anti-patterns)

- Do not hardcode the experimental connector list in Airbyte docs; Sonar registry must be the source of truth.
- Do not filter only `AgentConnectorRegistry`; sidebar and DR cross-links would still advertise hidden connectors.
- Do not infer availability from Airbyte replication registry/cloud registry; this is specifically Sonar direct/agentic availability.
- Do not silently fall back to all connector directories if registry fetch fails in CI or production build.
